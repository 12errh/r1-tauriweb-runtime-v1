## What v0.2 Fixes (The Root Cause)

Testing with `file-browser-tauri` revealed one systematic problem repeated
across every API package. Tauri apps use two different import patterns:

```typescript
// Pattern A — invoke() call (already works in v0.1)
const result = await invoke('read_text_file', { path: '/data.json' });

// Pattern B — direct import (broken in v0.1)
import { readTextFile } from '@tauri-apps/api/fs';
const result = await readTextFile('/data.json');
```

The Vite plugin rewrites Pattern B imports:
```
@tauri-apps/api/fs  →  @r1/apis/fs
```

But then Rollup looks for `readTextFile` as a named export in `@r1/apis/fs`
and finds nothing — because it only exists as a method on `FsPlugin`.
This causes a build-time crash on every app that uses Pattern B.

**The fix**: every API module needs both a KernelPlugin class (Pattern A)
AND top-level named exports (Pattern B) that call the same underlying logic.

## Phase 1 — Fix the Vite Plugin

> **Goal**: Fix the two bugs in `@r1/vite-plugin` that prevent any app
> from booting correctly. This is the highest-priority fix because it
> blocks everything else.

### Known Bugs (confirmed by file-browser-tauri test)

**Bug 1 — Wrong WASM file being loaded**
The plugin tells `loadBackend()` to load `module_name.js` (the JS glue)
instead of `module_name_bg.wasm` (the binary). `WebAssembly.instantiate()`
rejects anything that is not raw bytes.

**Bug 2 — Import patcher missing sub-paths**
The patcher likely only replaces `@tauri-apps/api` but not sub-paths:
```typescript
// Not being patched (sub-path imports):

import { homeDir } from '@tauri-apps/api/path'
import { listen } from '@tauri-apps/api/event'

import { invoke } from '@tauri-apps/api/core'
import { invoke } from '@tauri-apps/api/tauri'   // v1 format
```

### Agent Instructions

**Step 1.1 — Analyze the current WASM loading code**


**Step 1.2 — Fix the WASM path**
Change the extension from `.js` to `_bg.wasm`.

replaced by underscores.

**Step 2.2 — Implement the VFS singleton with promise lock**
At the top of the direct exports section, the singleton must use a
promise lock to prevent the race condition:

```typescript
let _vfs: VFS | null = null;
let _initPromise: Promise<VFS> | null = null;

async function getVfs(): Promise<VFS> {
  if (_vfs) return _vfs;
  if (!_initPromise) {
    _initPromise = (async () => {
      const instance = new VFS();
      await instance.init();
      _vfs = instance;
      return instance;
    })();
  }
  return _initPromise;
}
```

```typescript
export async function open(options?: OpenDialogOptions): Promise<string | string[] | null> {
  return new Promise((resolve) => {
    const input = document.createElement('input');
    input.type = 'file';
    if (options?.multiple) input.multiple = true;
    if (options?.directory) (input as any).webkitdirectory = true;
    if (options?.filters?.length) {
      input.accept = options.filters
        .flatMap(f => f.extensions.map(e => `.${e}`))
        .join(',');
    }
    input.onchange = () => {
      if (!input.files?.length) return resolve(null);
      const paths = Array.from(input.files).map(f => f.name);
      resolve(options?.multiple ? paths : paths[0]);
    };
    input.oncancel = () => resolve(null);
    input.click();
  });
}
```
